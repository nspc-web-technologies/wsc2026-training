## Introduction

We are going to create an albums management system for an office administrator (also known as "admin") to manage some song products. Song products belong to publishers. The publisher will also manage the associated album data as well. Admin can also view and manage these publishers and song products listing and editing.

Each song product record comes with a unique identifier number, we call it International Standard Recording Code (ISRC). An ISRC has 12 characters, usually written like this: `CC-XXX-YY-NNNNN` (e.g., `TW-XYZ-25-00821`), where:

- CC → Country code (e.g., TW for Taiwan)
- XXX → Registrant code (company/label code)
- YY → Year of registration (last 2 digits)
- NNNNN → Unique number for the specific recording

There are public-facing APIs. They are:

- User accessing functionalities
- Public album and song functionalities
- Other functionalities

## Public API

Anyone can query all album and song information in the Public API; no registration or login is required to use these functions.

Users can log in through the Login API. After logging in, they will receive an Access Token, which can be used to access API functions that require authorization. The Access Token is generated by MD5 hashing the user's username and converting it to a lowercase hex string.

### Cursor-Based Pagination

When the API requires pagination, it will use cursor-based pagination. Each returned result will include a `next_cursor` field. Users can pass the value of this field in the next API request to retrieve the next page of data. If `next_cursor` is `null`, it indicates that there is no more data available to query.

The value of `next_cursor` is a string obtained by taking the unique identifier of the last item from the previous result set, wrapping it in JSON, and then Base64 encoding it. By default, 10 records are returned per page, and users can adjust the number of records returned per page using the `limit` parameter.

Example: If there are 100 records with IDs from 1 to 100, and 10 records are returned per page, the first request will return data with IDs 1–10. It will also return `{"id":10}` (Base64 encoded) in the `next_cursor` field. When the user makes the next request, they will pass this string in the `cursor` parameter to retrieve data with IDs 11–20, and so on, until `next_cursor` is `null`.

### Year Filter

When fetching the album list API, the `year` parameter can be used to filter albums by a specific year. The `year` parameter supports a single year (e.g., `2020`) or a year range (e.g., `2018-2020`). If the `year` parameter is not provided, it will return albums from all years.

### Album Cover Logic

![Cover - 1 image](media/image5.jpeg) ![Cover - 2 images](media/image6.jpeg) ![Cover - 3 images](media/image7.jpeg)

The "Get Album Cover" logic is determined by the `songs.is_cover` property of the songs under that album. The cover is a combination of one, two, or three song `cover_images` (as shown in the image above).

The order of the combination must follow the order sequence of the songs within the album. If more than three songs [are marked as covers], an error message (`"Too many covers provided"`) will be returned.

When adding, modifying, deleting, or sorting songs, if it is related to the `is_cover` status, it will also affect the resulting cover combination.

Example: An album has five songs:

| Song  | is_cover | order |
|-------|----------|-------|
| Song1 | true     | 1     |
| Song2 | false    | 2     |
| Song3 | false    | 3     |
| Song4 | true     | 4     |
| Song5 | true     | 5     |

In this case, the album cover will be: a three-image combination composed of the covers from Song1, Song4, and Song5, in that specific order.

## User API

Users can log in to registered accounts via the Login API. When calling the API, the Access Token should be placed in the `X-Authorization` Header, allowing the API to perform user authentication via the access token.

When a user logs out, the Access Token they were using must be deleted (invalidated), preventing the user from logging in again [with that token].

If an Access Token is not provided when calling an API [that requires it], a 401 error status will be returned, along with a corresponding error message.

User capabilities:

- User can visit Song details
- User can Get Statistics Result

when params is:

- `?metrics=song`: Returns a list of songs, ordered by view count (desc).
- `?metrics=album`: Returns a list of albums, ordered by total view count (desc).
- `?metrics=label`: Returns a list of labels. Each label includes its top 10 songs, ordered by view count (desc).
  - (Optional) Can be combined with a `labels` parameter (e.g., `labels=Pop,Rock`) to filter for specific labels.

## Admin API

When a user's role is `"admin"`, they can manage other users, including retrieving all users, updating users, and blocking/unblocking users. Consequently, banned users are prevented from accessing the User APIs.

## Error Message

When an error occurs, an error message and HTTP status code must be returned according to the situation.

Error Response Format:

Response ( Unauthorized 401 )

Content-Type: `application/json`

```json
{
  "success": false,
  "message": "Access Token is required"
}
```

Error Message:

| Message | HTTP Status Code | Description |
|---------|-----------------|-------------|
| Access Token is required | 401 | Access Token not provided in `X-Authorization` header when calling a protected API. |
| Invalid Access Token | 401 | Provided an invalid, expired, or non-existent Access Token. |
| Access denied | 403 | A logged-in user attempts to operate on a resource that does not belong to them (e.g., updating or deleting another user's album). |
| Admin access required | 403 | A user whose role is not `'admin'` attempts to access an Admin API (e.g., `GET /api/users`). |
| User is banned | 403 | A user whose account has been blocked attempts to log in or access protected features. |
| Not Found | 404 | The requested API route does not exist. / The requested resource (e.g., Album, Song, User) does not exist (e.g., `GET /api/albums/999`). |
| Cover Not Found | 404 | The requested album or song cover image does not exist. |
| Too many covers provided | 400 | The operation (e.g., adding/modifying a song) results in the album having more than 3 cover images. |
| Login failed | 400 | Provided incorrect username or password during login. |
| Username already taken | 409 | The username is already taken (or: already in use) during registration. |
| Email already taken | 409 | The email is already taken (or: already in use) during registration. |
| Validation failed | 400 | Required fields are missing from the request (e.g., username not provided during registration). / A field in the request has an incorrect format (e.g., invalid email format, `release_year` is not a number). / When updating song order, `song_ids` contains invalid IDs or IDs that do not belong to the album. / Attempting to update a user's role to an invalid value. |
| Invalid parameter | 400 | Provided invalid query parameters, such as `limit` exceeding 100 or not being a number. |
| Invalid cursor | 400 | Provided a malformed or invalid cursor. |
| Invalid year format | 400 | The `year` parameter format is incorrect (e.g., `"abc"` or `"2000-1990"`). |
| Invalid file type | 400 | The uploaded `cover_image` is not an acceptable image format. |
| User not found | 404 | An admin attempts to operate on a non-existent `user_id`. |
| Cannot ban self | 400 | An admin attempts to block their own account. |
| Last admin demotion forbidden | 403 | The system must maintain at least one admin at all times. |
| Banned user update failed | 409 | Cannot update banned user role. |
| Cannot ban another admin | 403 | Admin cannot ban another admin. |
